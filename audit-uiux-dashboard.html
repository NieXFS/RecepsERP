Acessibilidade

CRITICAL — Prioridade 1

Critical KPI Cards sem landmarks semânticos e aria-labels aria-labels ›

Os 4 cards de KPI no Dashboard usam <Card> (que renderiza <div>) sem role, aria-label ou aria-labelledby. Para leitores de tela, esses blocos são apenas divs genéricos — o usuário não sabe que está em "Faturamento Hoje: R$ 1.250,00".

Antes (atual)
<Card> <CardHeader> <CardTitle className="text-sm font-medium text-muted-foreground"> Faturamento Hoje </CardTitle> <DollarSign className="h-4 w-4 text-muted-foreground" /> </CardHeader> <CardContent> <p className="text-2xl font-bold">R$ {kpis.revenue...}</p> </CardContent> </Card>
Depois (recomendado)
<Card role="region" aria-labelledby="kpi-revenue-label"> <CardHeader className="flex flex-row items-center justify-between pb-2"> <CardTitle id="kpi-revenue-label" className="text-sm font-medium text-muted-foreground"> Faturamento Hoje </CardTitle> <DollarSign className="h-4 w-4 text-muted-foreground" aria-hidden="true" /> </CardHeader> <CardContent> <p className="text-2xl font-bold" aria-live="polite"> R$ {kpis.revenue.toLocaleString(...)} </p> </CardContent> </Card>

Adicionar aria-hidden="true" nos ícones decorativos (DollarSign, TrendingUp, Calendar, UserPlus) para que não sejam lidos pelo screen reader.

High Botões de navegação de mês com touch target insuficiente touch-target-size ›

Os botões "icon-sm" (ChevronLeft/Right) para navegar entre meses têm size: 28px (h-7 / w-7). O mínimo recomendado pela Apple HIG é 44×44px e pelo Material Design 48×48dp.

Correção
// Aumentar para "icon" (32px) ou melhor "icon-lg" (36px) // e adicionar padding extra para área de toque: <Link className={cn(buttonVariants({ variant: "outline", size: "icon" }), "min-w-[44px] min-h-[44px]" )} aria-label="Ver mês anterior" > <ChevronLeft className="h-4 w-4" aria-hidden="true" /> </Link>
src/components/dashboard/monthly-summary-section.tsx
Medium KPI Cards não são clicáveis — oportunidade perdida de drill-down press-feedback ›

Os 4 cards de KPI são estáticos (divs). Em dashboards de ERP, o padrão esperado é clicar no card para ver detalhes. Ex: clicar "Agendamentos Hoje" levaria para /agenda?date=today.

Recomendação: Envolver cada card com <Link> e adicionar micro-interação de hover:

<Link href="/agenda?date=today" className="group"> <Card className="transition-all duration-200 group-hover:shadow-md group-hover:border-primary/20 group-hover:-translate-y-0.5" > ... </Card> </Link>

Isso adiciona: feedback visual no hover (shadow + lift), cursor pointer via Link, e affordance clara de que o card é interativo.

Performance & Loading

HIGH — Prioridade 3
3 findings
High Sem skeleton/loading state — CLS durante carregamento do Dashboard progressive-loading ›

A página Dashboard é um Server Component async que faz 3 chamadas paralelas (getDailyKPIs, getMonthlyStats, getDailyRevenue). Enquanto essas promises resolvem, o usuário vê... nada (ou o layout anterior). Não há loading.tsx com skeleton screens.

Medium Gráfico Recharts no bundle principal (sem lazy load) bundle-splitting ›

O componente MonthlyRevenueChart importa Recharts diretamente. Recharts é uma lib pesada (~250KB gzipped). Como o gráfico está abaixo do fold (após os KPI cards), ele deveria ser carregado com dynamic import.

// Em monthly-summary-section.tsx, trocar: import { MonthlyRevenueChart } from "@/components/dashboard/monthly-revenue-chart"; // Por: import dynamic from "next/dynamic"; const MonthlyRevenueChart = dynamic( () => import("@/components/dashboard/monthly-revenue-chart") .then(mod => mod.MonthlyRevenueChart), { loading: () => <div className="h-[320px] animate-pulse rounded bg-muted" />, ssr: false, } );
src/components/dashboard/monthly-summary-section.tsx
Low Navegação entre meses causa full server round-trip content-jumping ›

Clicar nos botões de mês anterior/próximo navega via <Link href="/dashboard?month=...">, causando uma nova requisição ao servidor. Isso gera flash de conteúdo (CLS). Considerar usar startTransition + router.push com estado de loading local, ou converter para client-side data fetching com SWR/React Query para a seção mensal.

High Sidebar fixa de 256px sem collapse — consome espaço em telas menores adaptive-navigation ›

A sidebar tem w-64 (256px) fixos e não possui mecanismo de collapse. Em telas de 1024px, o conteúdo principal fica com apenas ~700px. Não há breakpoint para esconder a sidebar em mobile — o layout usa flex h-screen sem tratamento responsivo.

Plano de ação
1. Mobile (< 768px): Sidebar como drawer/overlay com hamburger no Header 2. Tablet (768-1024px): Sidebar colapsada (apenas ícones, ~64px) 3. Desktop (> 1024px): Sidebar expandida (256px, como hoje) // Implementar com: // - Estado de collapse no layout (useState ou cookie) // - Sheet/Drawer do shadcn para mobile // - Botão de toggle no Header // - Transição suave: transition-[width] duration-300

Medium Gráfico não adapta em telas pequenas — eixo Y cortado responsive-chart ›

O MonthlyRevenueChart usa ResponsiveContainer para largura, mas o eixo Y tem width: 88px fixo para o formatador de moeda. Em telas estreitas (~375px menos sidebar), o eixo Y + margem + label pode consumir ~120px do espaço, deixando o gráfico espremido.

Recomendação
// Reduzir label do eixo Y em mobile: tickFormatter={(value) => { if (windowWidth < 640) { return `${(Number(value) / 1000).toFixed(0)}k`; } return currencyFormatter.format(Number(value)); }} // Ou usar margin dinâmica: margin={{ top: 8, right: 8, left: isMobile ? -12 : 4, bottom: 8 }}
src/components/dashboard/monthly-revenue-chart.tsx
Low Grid de KPI cards: gap inconsistente entre breakpoints spacing-scale ›

O grid usa gap-4 (16px) uniformemente. A seção mensal abaixo usa gap-4 também. Porém, o espaçamento entre seções é gap-6 (24px). Considerar usar gap-4 entre cards e gap-8 (32px) entre seções para criar hierarquia visual mais clara.

Tipografia & Cores

MEDIUM — Prioridade 6
3 findings
Medium Valores monetários não usam font tabular — layout shift em updates number-tabular ›

Os valores como "R$ 1.250,00" usam a fonte sans-serif padrão com figuras proporcionais. Quando os números mudam, o layout pode "pular" porque o "1" é mais estreito que o "8". Usar font-variant-numeric: tabular-nums nos valores financeiros resolve isso.

// Adicionar classe utilitária nos valores: <p className="text-2xl font-bold tabular-nums"> R$ {kpis.revenue.toLocaleString(...)} </p> // "tabular-nums" é uma classe nativa do Tailwind // Aplica font-variant-numeric: tabular-nums

Impacto: Todos os cards de KPI + cards mensais + eixo Y do gráfico.

src/app/(dashboard)/dashboard/page.tsx src/components/dashboard/monthly-summary-section.tsx
Medium Accent themes misturam OKLch com hex — inconsistência nos tokens color-semantic ›

O tema padrão (RECEPS_SIGNATURE) define todas as cores em oklch(), mas os accent themes alternativos (ROSE_ELEGANCE, EXECUTIVE_BLUE, etc.) usam hex puro (#b84773). Isso causa inconsistência de renderização: oklch é perceptualmente uniforme, hex não. Considerar converter todos os accent themes para OKLch.

src/app/globals.css
Low Chart colors: escala monocromática do roxo dificulta distinção color-guidance ›

As variáveis chart-1 a chart-5 formam uma escala do mesmo matiz roxo (285°). No gráfico que mostra "Faturamento", "Comissões" e "Despesas" simultaneamente, a distinção entre chart-1, chart-2 e chart-4 pode ser difícil, especialmente para pessoas com deficiência visual de cor.

Recomendação: Usar cores de matizes distintos para as 3 séries principais. Ex: chart-1 (roxo/primary para faturamento), chart-2 (teal/verde para comissões), chart-4 (amber/laranja para despesas). Manter a escala monocromática apenas para variações do mesmo dado.

Navegação

HIGH — Prioridade 9
2 findings
Medium Header não mostra breadcrumb ou contexto de página breadcrumb-web ›

O Header exibe apenas o nome do tenant à esquerda e controles do usuário à direita. Não há indicação de em que página o usuário está (essa info está no <h2> da própria página, abaixo do header). Para páginas com sub-níveis (ex: /configuracoes/aparencia), um breadcrumb no header seria valioso.

Low Logout sem confirmação — ação destrutiva sem dialog confirmation-dialogs ›

O botão "Sair" no header executa signOut() diretamente sem confirmação. Embora não seja uma ação tão destrutiva, em um ERP onde o usuário pode ter dados não salvos em formulários abertos, uma confirmação simples evita frustrações.

Charts & Data Visualization

LOW — Prioridade 10

Medium Legend do gráfico não é interativa legend-interactive ›

O <Legend /> do Recharts renderiza as 3 séries, mas clicar nelas não faz nada. Habilitar toggle de visibilidade ao clicar na legenda é padrão em dashboards e ajuda o usuário a focar em uma série específica.

// Adicionar estado para séries visíveis: const [hiddenSeries, setHiddenSeries] = useState<Set<string>>(new Set()); <Legend onClick={(e) => { const key = e.dataKey; setHiddenSeries(prev => { const next = new Set(prev); next.has(key) ? next.delete(key) : next.add(key); return next; }); }} /> <Line dataKey="faturamento" hide={hiddenSeries.has("faturamento")} ... />
src/components/dashboard/monthly-revenue-chart.tsx
Low Tooltip do gráfico sem acessibilidade via teclado tooltip-keyboard ›

O tooltip do Recharts só aparece com hover do mouse. Não há como navegar pelos data points com teclado. Isso é uma limitação do Recharts em si, mas pode ser mitigado com tabIndex nos activeDots ou com o Accessibility module do Recharts.
